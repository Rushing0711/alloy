# 技能编写指南

> 写出 Agent 参照后能开发出优秀技能的指南——平台规范是基础，最佳实践是提升，开发流程是保障。

---

## 一、技能类型与质量标准

技能不是一种东西。不同类型的技能有不同的写法、不同的说服策略、不同的验证方法。写之前先确定类型。

### 四种技能类型

| 类型 | 特征 | 写法 | 说服策略 | 验证 | 例子 |
|------|------|------|---------|------|------|
| **闸门型** | 二元判断，通过/不通过 | 绝对语言，无例外条款 | 权威 + 承诺 + 社会认同 | 压力场景 | 阶段校验、hash 验证、分支位置检查 |
| **路径型** | 需要判断力，场景多变 | 解释 WHY，给方向不给细节 | 适度权威 + 团结 | 应用场景 | 需求设计、代码审查、brainstorming |
| **参考型** | 提供 API/语法/模式知识 | 检索友好，结构化，Gotchas | 仅清晰度 | 检索场景 | API 文档、语法参考、配置指南 |
| **心智型** | 改变思考框架 | 模式识别 + 应用场景 | 承诺 + 社会认同 | 识别+应用 | TDD 思维、系统化调试 |

### 说服策略按类型分治

基于 Cialdini (2021) 和 Meincke et al. (2025) 的研究，说服技术将 LLM 遵从率从 33% 提升到 72%。但不是所有说服原则都适用于所有技能：

| 说服原则 | 闸门型 | 路径型 | 参考型 | 心智型 |
|---------|--------|--------|--------|--------|
| 权威（命令式语言） | ✅ 必需 | ⚠️ 适度 | ❌ | ❌ |
| 承诺（显式选择/公告） | ✅ 必需 | ✅ 推荐 | ❌ | ✅ 推荐 |
| 社会认同（普遍/失败模式） | ✅ 必需 | ⚠️ 适度 | ❌ | ✅ 推荐 |
| 喜好 | ❌ | ❌ | ❌ | ❌ |
| 互惠 | ❌ | ❌ | ❌ | ❌ |

刻意不用的原则：**喜好**（与诚实反馈冲突，产生阿谀奉承）、**互惠**（操控感强，技能场景很少需要）。

### 质量标准

每种类型有不同的"好"的定义：

- **闸门型**：走过场不可能——操作本身就是约束的执行，Agent 没有裁量空间
- **路径型**：方向正确不僵化——Agent 能在变化场景中做出合理判断
- **参考型**：检索即答案——找到信息就能直接使用
- **心智型**：识别即应用——看到场景就知道用什么框架处理

## 二、平台规范

技能运行在 Claude Code 的技能系统中。了解平台的规则和限制，才能写出正确触发的技能。

### 2.1 Frontmatter 字段

SKILL.md 由 YAML frontmatter（`---` 标记之间）+ Markdown 正文组成。

**必需字段：**

| 字段 | 说明 | 约束 |
|------|------|------|
| `name` | 显示名称 | 字母+数字+连字符，1-64 字符，必须匹配目录名 |
| `description` | 触发条件描述 | 只写"何时用"，不写"做什么"。与 `when_to_use` 合计 ≤ 1536 字符 |

**常用字段：**

| 字段 | 说明 | 默认值 |
|------|------|--------|
| `when_to_use` | 额外触发上下文（短语、示例请求），附加到 description | — |
| `disable-model-invocation` | 阻止 Claude 自动加载，仅手动 `/name` 触发 | `false` |
| `user-invocable` | 从 `/` 菜单隐藏，仅 Claude 可调用 | `true` |

**特殊场景字段：**

| 字段 | 说明 |
|------|------|
| `arguments` | 命名位置参数，支持 `$name` 替换 |
| `argument-hint` | 自动补全参数提示，如 `[issue-number]` |
| `allowed-tools` | 技能激活时免确认的工具列表 |
| `disallowed-tools` | 技能激活时移除的工具列表 |

**高级字段：**

| 字段 | 说明 |
|------|------|
| `model` | 覆盖使用的模型 |
| `effort` | 推理力度：`low`/`medium`/`high`/`xhigh`/`max` |
| `context` | 设为 `fork` 在隔离子代理中运行 |
| `agent` | `context: fork` 时使用的子代理类型（`Explore`/`Plan`/`general-purpose` 或自定义） |
| `hooks` | 技能生命周期钩子 |
| `paths` | Glob 模式，限制技能激活的文件范围 |
| `shell` | Shell 命令使用的 shell：`bash`（默认）或 `powershell` |

#### 调用控制矩阵

| Frontmatter 设置 | 用户可调用 | Claude 可调用 | description 在上下文中 |
|-----------------|-----------|-------------|---------------------|
| 默认 | 是 | 是 | 是 |
| `disable-model-invocation: true` | 是 | 否 | 否 |
| `user-invocable: false` | 否 | 是 | 是 |

#### 按技能类型推荐字段组合

| 技能类型 | 推荐字段 |
|---------|---------|
| 闸门型 | `name` + `description` + `user-invocable: false`（仅 Claude 自动调用，确保拦截） |
| 路径型 | `name` + `description` + `when_to_use` |
| 参考型 | `name` + `description` + `when_to_use` + `paths` |
| 心智型 | `name` + `description` + `when_to_use` |

### 2.2 目录结构与作用域

**存放位置与作用域：**

| 位置 | 路径 | 作用域 |
|------|------|--------|
| 个人级 | `~/.claude/skills/<name>/SKILL.md` | 你所有项目 |
| 项目级 | `.claude/skills/<name>/SKILL.md` | 仅此项目 |
| 插件级 | `<plugin>/skills/<name>/SKILL.md` | 插件启用范围（`plugin-name:skill-name` 命名空间） |

优先级：企业 > 个人 > 项目。同名 skill 优先于 command。

**辅助文件目录：**

```
skill-name/
├── SKILL.md           # 主指令文件（必需）
├── templates/         # 输出模板（可选）
├── references/        # 参考文档（可选）
├── scripts/           # 可执行脚本（可选）
├── examples/          # 示例输出（可选）
└── assets/            # 静态资源（可选）
```

### 2.3 渐进式披露

技能使用三级加载系统：

1. **发现阶段**（~100 token/技能）：启动时仅加载 `name` + `description`
2. **激活阶段**（<5000 token 推荐，SKILL.md < 500 行）：任务匹配时加载完整 SKILL.md 正文
3. **执行阶段**（按需）：按需加载 `references/`、`scripts/`、`templates/` 中的文件

设计含义：`description` 是发现的唯一入口，正文是激活的核心内容，辅助文件是深度的按需资源。

**三种渐进式披露模式：**

1. **高层指南 + 引用** — SKILL.md 提供快速入门，详细内容在外部文件
2. **按领域组织** — 不同领域放不同参考文件（如 `references/aws.md`、`references/gcp.md`），Claude 只读取相关文件
3. **条件性细节** — 基本内容内联，高级特性链接到外部文件

关键规则：**引用只深一层**。SKILL.md 直接链接到参考文件，参考文件不应再链接到更深层文件。长参考文件（>300 行）顶部加目录。

### 2.4 动态上下文注入

`` !`command` `` 语法：技能内容发送给 Claude 之前执行 shell 命令，输出替换占位符。

```markdown
## 当前 PR 上下文
- PR diff: !`gh pr diff`
- PR 状态: !`gh pr view --json state,mergeable`
```

多行命令使用 ` ```! ` 围栏代码块。

注意：命令输出作为纯文本插入，不会再次扫描 `!` 占位符。`!` 必须出现在行首或空白之后才被识别。

### 2.5 字符串替换变量

| 变量 | 说明 |
|------|------|
| `$ARGUMENTS` | 调用时传入的所有参数 |
| `$ARGUMENTS[N]` / `$N` | 按索引访问参数（0-based），如 `$0`、`$1` |
| `$name` | frontmatter `arguments` 中声明的命名参数 |
| `${CLAUDE_SESSION_ID}` | 当前会话 ID |
| `${CLAUDE_EFFORT}` | 当前推理力度 |
| `${CLAUDE_SKILL_DIR}` | 技能 SKILL.md 所在目录 |

用反斜杠转义：`\$1.00` 保留字面 `$`。

### 2.6 技能内容生命周期

- 技能被调用后，渲染后的内容在**会话剩余期间一直保留**
- 上下文压缩时，每个被调用的技能保留前 **5000 token**，多个技能共享 **25000 token** 预算
- 如果技能似乎不再影响行为，通常是模型选择了其他方法，而非内容丢失

## 三、编写原则

12 条原则，统一 writing-skills 和 skill-creator 的共识与互补。核心逻辑：闸门型用绝对语言，其他类型解释 WHY。

### 1. 解释 WHY 优于堆砌 MUST

writing-skills 和 skill-creator 的共识：解释原因比强制指令更有效。今日 LLM 足够聪明，理解 WHY 后能做出更好的判断。

**闸门型是例外**——闸门需要绝对语言（"你必须"、"没有例外"），因为闸门的本质就是消除裁量空间。

```markdown
# ❌ 大写禁令堆砌——指令疲劳
MUST 至少 3 轮追问。DO NOT 在 1-2 轮后进入方案阶段。

# ✅ 解释 WHY——Agent 理解后自觉遵守
每次只问一个问题，基于回答逐步深入。你的目标不是"问够 3 个问题"交差——
而是真正理解用户要什么、不要什么、怎么算成功。

# ✅ 闸门型——绝对语言是合理的
先写失败的测试，再写生产代码。没有例外。
写代码前没有测试？删掉。从头开始。
```

**判断标准：** 逻辑是二元判断（通过/不通过）→ 绝对语言；需要权衡判断 → 解释 WHY。

### 2. 说服科学三原则

基于 Cialdini (2021) 和 Meincke et al. (2025) 研究，说服技术将 LLM 遵从率从 33% 提升到 72%（p < .001, N=28,000）。

**权威（Authority）**：命令式语言消除决策疲劳和合理化空间。主要用于闸门型。

**承诺（Commitment）**：要求显式选择和公告，使 Agent 对流程做出承诺。

```markdown
# ✅ 承诺机制
开始前宣布："我将执行 [阶段名]，目标 [具体目标]"
完成后报告："[阶段名] 完成，证据 [具体结果]"
```

**社会认同（Social Proof）**：用普遍模式和失败模式强化规范。

```markdown
# ✅ 普遍模式
每次修改代码后都必须运行测试——这是可靠工作流的基本要求

# ✅ 失败模式
没有失败测试的生产代码 = 未验证的假设
```

### 3. 抗合理化三层防御

闸门型技能必须包含三层防御。Agent 会在压力下合理化绕过规则，三层防御逐层拦截：

**第一层：显式否定** — 列出"无例外"条款，切断"这次可以"的借口

```markdown
写代码前没有测试？删掉。从头开始。
无例外：
- 不要保留作为"参考"
- 不要"边写测试边适配"
- 不要看它
- 删掉就是删掉
```

**第二层：精神/字面阻断** — 一句话切断"我在遵循精神"的借口

```markdown
违反字面规则就是违反精神规则。
```

第二层是抗合理化三层中**最容易被作者跳过、最容易被 agent 钻空的一层**——agent 知道"不能跳过测试"（第一层），知道"不能用'紧急'当借口"（第三层 Red Flags），但当面对"先生成框架代码再补测试，反正最终也是 TDD"这种**字面变通但精神保留**的情况，没有第二层就会破防。

**适用场景（满足任一即必须有第二层）：**

1. **agent 在历史 transcript 中反复"换说法"绕过同一规则** — 第一层禁令容易被同义改写攻破，需要第二层兜底
2. **规则有"精神 vs 字面"歧义可乘** — 例如"先写测试再写代码"在 agent 眼里可能等于"测试和代码同时存在但测试先于 commit"
3. **在压力场景下需要最强抗辩** — 时间/沉没成本/权威压力会触发"务实精神 > 字面规则"的合理化

**标准措辞模板：**

| 模板 | 适用场景 | 示例 |
|------|---------|------|
| `违反字面 = 违反精神` | 字面规则即精神规则的体现 | "先写失败的测试，再写生产代码——违反字面就是违反精神" |
| `绕过流程 = 不算流程` | 流程必经节点不可"等价替代" | "审查跳过就是没审查——再快的检查不算审查" |
| `延后 = 不算执行` | 操作有时序依赖 | "归档前未提交就是未提交——'后面会提交'不算已提交" |

措辞要求：一句话，重复关键词形成对仗，等号或破折号构造视觉对照。

**反例（什么算没写第二层）：**

- ❌ `"请严格遵守 TDD"` — 这是重复第一层，没有第二层的对照结构
- ❌ `"任何变通都是违规"` — 抽象，没指明字面与精神的关系
- ❌ `"如果你想绕开规则，请先和用户确认"` — 把禁令换成了 USER_GATE，等于消解了规则本身
- ❌ `"测试不是形式主义，要真心写"` — 诉诸态度而非规则，agent 可以承认"我真心在做"然后继续违规

**第三层：Red Flags 自检** — 列出 Agent 常见的合理化思维

```markdown
如果你发现自己这么想：
- "只是改个错别字，直接编辑吧"
- "用户很急，跳过审查吧"
- "这个太简单了，不需要测试"
→ 这些意味着：停下来。回到流程。
```

Red Flags 表（借口 vs 现实对照）是第三层的一种具体形式：

```markdown
| 借口 | 现实 |
|------|------|
| "只是改个错别字" | 直接编辑绕过了完整性检查 |
| "用户很急" | 急不是绕过流程的理由 |
```

**三层协作：**

```
第一层（显式无例外）  → 切断"这次可以"的借口
第二层（精神=字面）   → 切断"我在遵循精神"的借口
第三层（Red Flags）   → 切断"我现在的想法听起来合理"的借口
```

三层覆盖三类合理化路径——例外、变通、借口——缺一层就有漏洞。

### 4. 四类语义节点术语

闸门型和路径型技能里频繁出现"卡点"——前置失败、绝对禁令、用户决策、软警告，性质完全不同。用统一术语命名这四类节点，让 skill 作者写得清、agent 读得对、frontmatter 数得准。

**术语对照：**

| 术语 | 触发主体 | 含义 | Agent 行为 | 违反后果 |
|------|---------|------|-----------|---------|
| `PRECONDITION_FAIL` | 系统/状态 | 前置条件校验失败（环境、phase、依赖） | 立即中止本次会话，输出修复指引 | 不可继续——继续=带病执行 |
| `HARD_STOP` | 对 Agent 的绝对禁令 | 无论看似多合理 agent 都不许跨越的边界 | 拒绝执行，回到流程或终止 | 违反 Iron Law |
| `USER_GATE` | 对用户的硬交互 | 必须用户决策才能继续（不可逆/方向选择/资源占用） | 调用平台问答工具等待用户输入 | 沉默 ≠ 授权 |
| `WARN` | 对当前会话的软提示 | 风险存在但不阻断 | 提示并记录，继续执行 | 仅留痕 |

**与四种技能类型的映射：**

| 技能类型 | 主用节点 | 辅用节点 |
|---------|---------|---------|
| 闸门型 | PRECONDITION_FAIL + HARD_STOP | USER_GATE（不可逆操作） |
| 路径型 | USER_GATE + WARN | HARD_STOP（极少数硬约束） |
| 参考型 | — | WARN（gotcha 提示） |
| 心智型 | — | — |

**与三层防御的关系：**

`HARD_STOP` 是第 3 节第一层"显式无例外条款"的具体形式之一。Iron Law 通常对应一条最核心的 HARD_STOP，被提升到顶部代码块；其他 HARD_STOP 嵌入步骤就近位置。

**关键区分（最常被混淆）：**

```
PRECONDITION_FAIL  vs  HARD_STOP
────────────────       ───────────────
触发：进入 skill 时    触发：执行过程中
原因：环境/状态不对    原因：agent 想跨越边界
对象：用户应修复       对象：agent 应停手
```

```
HARD_STOP  vs  USER_GATE
─────────       ──────────
拒绝者：agent   决策者：用户
"agent 不许做"  "用户必须选"
单向禁令         双/多选交互
```

**反例（什么算混用）：**

- ❌ `"phase ≠ archived → HARD STOP"` — 这是前置条件失败，应是 `PRECONDITION_FAIL`，措辞应是"环境不满足，请运行 X 修复"，而非"违反 Iron Law"
- ❌ `"merge 冲突时禁止 git merge --abort，🔴 STOP"` — 禁令对象是 agent 不是用户，应是 `HARD_STOP`；用户决策"放弃/保留"才是后续的 `USER_GATE`
- ❌ `"git pull 失败 → ⚠️ WARN 继续"` — 基础过期会污染下游 merge，这是阻断条件，应升 `PRECONDITION_FAIL` 或 `USER_GATE`，不是 `WARN`

**写法约定：**

- 在 skill 正文里用 `[PRECONDITION_FAIL]` / `[HARD_STOP]` / `[USER_GATE]` / `[WARN]` 标记节点起点（项目可自行映射到 emoji 视觉规范）
- frontmatter 若有节点统计字段，按四类分别计数，禁止合并
- 同一节点不要兼具两类语义（HARD_STOP 之后立刻 USER_GATE 是合法的串联，但单个节点不能既是禁令又是询问）

### 5. 通用禁令模式

**为什么需要"通用禁令"：** 跨多个 skill 反复出现的同款隐患，应该抽成共享禁令清单单独维护——否则每个 skill 作者都要独立思考"agent 看到冲突会不会自己跑 abort"，且容易遗漏。

通用禁令的特征：
- **触发场景跨多个 skill 重现**（不是某 skill 特有）
- **agent 出于"让流程干净"的本能会主动执行**（不是用户明确要求）
- **执行后用户已 stage 的工作不可恢复**（破坏性强）

通用禁令清单是**抗合理化工具的延伸**——三层防御教 skill 作者"把规则写进 skill 里"，禁令清单告诉作者"这些常见破坏性命令必须列入禁令"。

#### 5.1 git 自救命令禁令

**触发场景：** 任何会出现 merge / rebase / pull 冲突或意外 dirty 状态的 skill。包括：
- 分支合并（squash、no-ff、fast-forward）
- worktree 创建/合并/清理
- 主分支同步（`git pull` 失败、远端分歧）
- 状态恢复（HEAD 偏移、文件冲突）

**禁令清单：**

| 命令 | agent 的合理化 | 真实后果 |
|------|--------------|---------|
| `git merge --abort` | "让流程干净地失败，重来一次" | 用户已 stage 的冲突解决方案被冲掉 |
| `git rebase --abort` | "rebase 出错了，回到起点重做" | 中途已解决的冲突全部丢失 |
| `git reset --hard` | "工作树状态不对，回到干净状态" | 未提交的所有改动消失，包括用户刚写的代码 |
| `git checkout .` | "把这些意外改动撤销掉" | 工作树未提交改动全部丢失 |
| `git restore .` | 同上的现代写法 | 同上 |
| `git restore --staged .` | "暂存区清空重新来" | 用户精心 stage 的部分提交计划消失 |
| `git stash` / `git stash drop` | "先 stash 一下让流程继续" | stash 容易被遗忘 / drop 直接丢工作 |
| `git clean -fd` | "把这些 untracked 文件清掉" | 用户尚未 add 的新文件消失 |
| `git push --force` / `--force-with-lease` | "远端分歧了，覆盖一下" | 别人推送的提交被覆盖 |

**WHY：** 这些命令的共同特征——**agent 看到失败状态时为"让 skill 流程能继续"会主动跑**，但每一个都会清空"用户视为有价值的当前状态"。Skill 流程退出比"清理状态再继续"更安全——退出后用户至少能看到现场。

**标准措辞模板：**

```
[HARD_STOP] 冲突/失败时禁止运行 git merge --abort、git rebase --abort、
git reset --hard、git checkout .、git restore .、git stash、git clean -fd、
git push --force 任何一个。

违反字面 = 违反精神：哪怕看似"只清理一下让流程继续"，也算违反禁令——
退出 skill 让用户处理是唯一合法路径。

冲突时的合法行为：报告冲突现场（git status / git diff），调用 USER_GATE 让
用户决定保留方案/放弃方案/手动解决，等待用户输入后再继续。
```

**嵌入位置：**

不是放在 skill 顶部 Iron Law 旁——禁令需要**就近嵌入触发命令的步骤里**。例如：
- merge / rebase / pull 命令前一行
- worktree 创建 / 合并 / 清理步骤的注释里
- 任何 `git commit` 失败处理分支里

集中放置的禁令在 agent 执行到某步时已不在注意力范围。

#### 5.2 扩展占位

后续累积的通用禁令（如 memory 批量写入、状态推进早于实际操作等）按 5.1 的格式继续添加。每条禁令包括：触发场景 / 禁令清单 / WHY / 标准措辞 / 嵌入位置五要素。

### 6. Gate Function 伪代码

对关键决策点，用伪代码消除歧义。比自然语言更精确，比脚本更灵活。

```markdown
BEFORE claiming completion:
1. IDENTIFY: 什么命令能证明这个断言？
2. RUN: 执行完整命令
3. READ: 读取完整输出，检查退出码
4. VERIFY: 输出是否确认断言？
   - NO: 陈述实际状态 + 证据
   - YES: 陈述断言 + 证据
5. ONLY THEN: 做出断言
```

适用场景：完成声明、质量闸门、状态转换。**不适用于路径型判断**——那种用方向不用伪代码。

### 7. 泛化而非过拟合

不要为了几个测试用例加 fiddly 的规则，而要找更好的模式和比喻。如果某个问题顽固地出现，尝试不同的隐喻或工作模式，而不是加更多约束。

```markdown
# ❌ 过拟合——为特定案例加 ad-hoc 规则
如果用户提到 TypeScript 配置文件且 tsconfig.json 存在且 node_modules 缺失，
则先执行 npm install 再检查 tsconfig...

# ✅ 泛化——找更好的模式
修改配置文件后，验证依赖是否完整：运行项目的安装命令，然后运行类型检查
```

### 8. 保持精简

移除不生效的内容。读 transcripts（不仅是最终输出）——如果 Agent 在无效步骤上浪费时间，说明指令太模糊、不适用、或选项太多。删掉不拉动的部分。

提供默认值而非菜单——选择一个默认方案，简要提及替代方案。

### 9. 流程图使用规范

**何时用流程图：**
- 非显而易见的决策点
- 循环可能过早停止的地方
- "何时用 A vs B"的选择

**何时不用流程图：**
- 参考材料 → 用表格
- 代码示例 → 用 Markdown 代码块
- 线性指令 → 用编号列表

**Graphviz 形状约定：**

| 形状 | 语义 | 例子 |
|------|------|------|
| `diamond` | 问题/决策 | "测试通过了吗？" |
| `box` | 动作 | "编写测试" |
| `plaintext` | 命令 | "`npm test`" |
| `ellipse` | 状态 | "测试失败" |
| `doublecircle` | 入口/出口 | "过程开始" |
| `octagon` + 红色填充 | 警告 | "从不忽略错误" |

**标签命名约定：**
- 问题以 `?` 结尾："Should I do X?"
- 动作以动词开头："Write the test"
- 命令使用字面值："`grep -r 'pattern' .`"
- 状态描述情形："Test is failing"

### 10. 引用强度标记

技能间引用使用三档强度，让 Agent 清楚知道引用是必需还是可选：

| 强度 | 格式 | 含义 |
|------|------|------|
| **REQUIRED SUB-SKILL** | `Use plugin:skill-name` | 必须使用，是流程的一部分 |
| **REQUIRED BACKGROUND** | `Understand plugin:skill-name` | 必须理解，不一定在当前流程调用 |
| **Related** | `See also: skill-name` | 相关但可选 |

**引用格式规则：**
- 使用命名空间前缀 + 技能名，**不使用文件路径**
- **不使用 @ 链接**（`@` 语法会强制立即加载文件，在需要前消耗 context）

```markdown
# ❌ 文件路径引用——不清晰是否必需
See skills/testing/test-driven-development

# ❌ @ 链接——强制加载，烧 context
@skills/testing/test-driven-development/SKILL.md

# ✅ 命名空间引用——清晰、按需加载
**REQUIRED SUB-SKILL:** Use superpowers:test-driven-development
```

### 11. 捆绑重复逻辑为脚本

如果多个测试用例中 Agent 都独立写了相同的辅助逻辑，说明技能应该自带这个脚本。写一次放 `scripts/`，告诉技能使用它，避免每次调用重新发明。

### 12. 用反例定义边界

光说"做得好"没用——Agent 不知道边界在哪。给出具体反例，Agent 就知道怎么避免了。

```markdown
什么算"不够"（反例）：
- 只问了一个选择题（"A. Web 应用 B. CLI"）就觉得自己完成了
- 用户说"Web 应用"后没有追问"单页还是多页？需要后端吗？"
```

## 四、技能结构规范

### 4.1 Iron Law

每个技能开头一条不可违反的铁律，代码块格式，2-3 行说清。从该技能的抗合理化防御中提炼最核心的一条。

```markdown
`NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST`
先写失败的测试，再写生产代码。没有例外。
```

```markdown
`NO DIRECT EDITING OF GENERATED ARTIFACTS`
已生成制品禁止直接编辑——哪怕错别字也必须重新生成
```

### 4.2 技能骨架

```
[顶部]  Iron Law（2-3 行代码块）+ Phase Overview
[主体]  Step 1 → Step 2 → Step 3
        每步自包含：验证门 + 操作 + 就近约束
[底部]  Common Rationalizations 表（Red Flags 表的一种形式）
```

**关键：全局约束不集中在顶部/底部，拆散嵌入各步骤的验证门。** Agent 执行到某步时，约束就在那步里。集中放置的约束不在 Agent 的注意力范围内。

### 4.3 Frontmatter 按类型推荐

| 技能类型 | 推荐字段 | 理由 |
|---------|---------|------|
| 闸门型 | `name` + `description` + `user-invocable: false` | 仅 Claude 自动调用，确保拦截不会遗漏 |
| 路径型 | `name` + `description` + `when_to_use` | 补充触发上下文 |
| 参考型 | `name` + `description` + `when_to_use` + `paths` | 限制激活的文件范围，避免误触发 |
| 心智型 | `name` + `description` + `when_to_use` | 补充触发上下文 |

### 4.4 技能链设计

每个技能的终端状态应明确指向下一个技能，形成线性工作流链而非网络：

```markdown
# ✅ 线性链条——每个技能有明确的后继
brainstorming → writing-plans → executing → finishing
```

```markdown
# ❌ 网状依赖——技能间循环，Agent 不知走哪条路
brainstorming ↔ planning ↔ reviewing
```

**设计规则：**
- 每个技能的终端状态只能调用**一个**后续技能
- 禁止循环引用（A 调用 B，B 又调用 A）
- 并行分支由调度型技能管理，不由技能自身管理

### 4.5 目录组织

```
skill-name/
├── SKILL.md           # 主指令文件（骨架 + 就近约束）
├── references/        # 参考文档（按需加载）
├── scripts/           # 可执行脚本（确定性任务）
├── templates/         # 输出模板
├── examples/          # 示例输出
└── assets/            # 静态资源
```

辅助文件引用最多一层深：SKILL.md → 子文件可以，SKILL.md → 子文件 → 孙文件不行。

## 五、技能开发流程

统一 writing-skills 和 skill-creator 的方法论为一个连贯流程。两个方法论的控制风格看似冲突（绝对命令 vs 解释 WHY），实际按技能类型分治后完全互补：

- 闸门型：writing-skills 的 TDD + 抗合理化装甲
- 路径型：skill-creator 的解释 WHY + 泛化精简
- 参考型：数据驱动迭代
- 心智型：识别+应用场景验证

### 5.1 捕获意图

确定三件事：

1. **技能做什么** — 它要解决什么问题？
2. **何时触发** — 用户会说什么话、处于什么上下文？
3. **输出格式** — 期望的结果长什么样？

同时确定技能类型（闸门/路径/参考/心智），这决定后续所有步骤的控制风格。

检查可用 MCP——如果有用（搜索文档、找相似技能、查最佳实践），并行调研减少用户负担。

### 5.2 基线测试

**先看失败，再写技能。** 但"失败"的观察方式取决于场景——新 skill 和成熟 skill 改造方法不同。

#### 5.2.1 新 skill 的基线测试（writing-skills RED）

无技能时让 Agent 执行任务，记录：

- 它跳过了什么步骤？
- 它用什么借口合理化绕过？（逐字记录——这些措辞后面用来写 Red Flags）
- 什么压力触发了违规？

#### 5.2.2 成熟 skill 改造的基线测试

**问题：** 当 skill 已经在生产中运行（如多阶段命令链），无法剥离单个 skill 做"无技能基线"——用户依赖完整链路工作，剥离会破坏其他流程。

**改造场景的三种基线手段（组合使用，互补不互斥）：**

##### 手段 1：节点审计

按第 3 章 §4 四类术语（PRECONDITION_FAIL / HARD_STOP / USER_GATE / WARN）逐条列出现有 skill 的所有节点，对账 frontmatter 标称数字。

**对账输出格式：**

```
skill: archive.md
─────────────────
节点统计（实际）:
  PRECONDITION_FAIL: 3 (line 60, 67, 74)
  HARD_STOP:         9 (line 41, 88, 102, ...)
  USER_GATE:         3 (line 97, 117, 159)
  WARN:              0

frontmatter 标称:
  hard_stops: 2     ❌ 偏离 -7
  stops: 3          ✅ 与 USER_GATE 对齐

节点缺失（应有未有）:
  - merge 冲突时禁止 git --abort 类命令（HARD_STOP）
  - memory 写入逐条确认（USER_GATE，当前是批量）
```

**节点审计的产出：** 偏离清单 + 缺失清单。这两份清单直接成为重写计划的输入。

##### 手段 2：压力场景重放

不剥离 skill，在**现有 skill 完整运行的前提下**，用 §5.4 的压力组合让 agent 执行任务，记录违规：

| 压力组合 | 关注的违规 |
|---------|-----------|
| 时间 + 沉没成本 | "已经做了这么多，merge 冲突 abort 重来太亏" → 自动 abort |
| 权威 + 务实 | "用户说尽快合，精确字符串确认是形式主义" → 跳过精确确认 |
| 疲惫 + 社交 | "memory 条目都看着合理，逐条问太啰嗦" → 批量写入 |

**与新 skill 基线的差异：** 新 skill 看的是"完全无约束下 agent 怎么失败"，成熟 skill 看的是"现有约束下 agent 还能怎么钻空"——两者都重要，后者揭示当前 skill 的防御漏洞。

##### 手段 3：跨 skill 链路追踪

同一隐患如果跨 ≥2 个 skill 重现 → 应进第 3 章 §5 通用禁令模式，而不是在各 skill 里重复写。

**追踪方法：**

1. 把手段 1 产出的"缺失清单"按主题归类（git 操作类 / 状态推进类 / 用户交互类）
2. 同主题在 ≥2 个 skill 出现 → 抽成通用禁令
3. 通用禁令进 §3.5，各 skill 按"嵌入位置"指引引用

**例：** archive.md 的"merge 冲突 abort"和 finish.md 的"squash 冲突 abort"是同款隐患 → 升级为 §3.5.1 git 自救禁令，而不是两个 skill 各写一遍。

#### 5.2.3 三种手段的关系

```
手段 1（节点审计）   → 找出"已写但写错"的节点（frontmatter 偏离）
手段 2（压力重放）   → 找出"未写但应写"的节点（现有防御漏洞）
手段 3（链路追踪）   → 找出"应共享但散写"的节点（升级为通用禁令）
```

三者并行做完，得到完整的改造任务列表：偏离修正 + 漏洞补防 + 通用禁令抽取。然后进入 §5.5 迭代改进。

### 5.3 编写草稿

针对基线观察到的具体违规写技能，不预设 Agent 会犯什么错。

**按类型选择控制风格：**

| 技能类型 | 控制风格 | 来源 |
|---------|---------|------|
| 闸门型 | 绝对语言 + 抗合理化三层防御 | writing-skills |
| 路径型 | 解释 WHY + 给方向不给细节 | skill-creator |
| 参考型 | 结构化 + Gotchas + 输出格式模板 | 两者共识 |
| 心智型 | 模式识别场景 + 应用场景 | writing-skills |

### 5.4 测试评估

**writing-skills 的定性 + skill-creator 的定量，融合使用：**

**压力场景（定性，来自 writing-skills）：**

| 压力类型 | 触发语 | 对应合理化 |
|---------|--------|-----------|
| 时间 | 紧急、截止日期、部署窗口 | "来不及了，跳过吧" |
| 沉没成本 | 数小时工作、删除浪费 | "已经做了这么多了，不值得重来" |
| 权威 | 高级说跳过、经理覆盖 | "他们比我懂，听他们的" |
| 经济 | 工作、晋升、公司存亡 | "不交付就完蛋了" |
| 疲惫 | 下班时间、已经很累 | "差不多就行了吧" |
| 社交 | 显得教条、不灵活 | "别人都不这样" |
| 务实 | 务实 vs 教条 | "务实一点" |

最佳压力测试组合 **3+ 种压力**。

**基线对比 + 定量评分（定量，来自 skill-creator）：**

并排运行 with-skill 和 without-skill（或 old-skill），对比输出质量。主观技能（写作风格、设计质量）用定性审查，客观技能（文件变换、数据提取、固定工作流）用定量断言。

**技能类型差异化验证：**

| 技能类型 | 验证方式 | 成功标准 |
|---------|---------|---------|
| 闸门型 | 压力场景——组合 3+ 种压力 | 最大压力下仍遵守 |
| 路径型 | 应用场景——变体场景 + 信息缺失 | 成功应用到新场景 |
| 参考型 | 检索场景——能否找到正确信息 | 找到并正确使用 |
| 心智型 | 识别+应用场景 + 反例 | 正确识别何时/如何应用 |

### 5.5 迭代改进

**writing-skills 堵漏洞 + skill-creator 泛化精简，互补使用：**

**堵漏洞（writing-skills）：** Agent 找到新的合理化借口？添加 4 层防御：
1. 显式否定规则
2. 合理化表格条目
3. Red Flags 条目
4. 更新 description

**泛化精简（skill-creator）：**
- 不要为特定案例加 ad-hoc 规则，找更好的模式和比喻
- 移除不生效的内容——读 transcripts 发现浪费时间的指令
- 提供默认值而非菜单

**循环直到：**
- Agent 在最大压力下仍遵守规则（闸门型）
- Agent 成功应用到新场景（路径型）
- 用户对输出满意
- 不再做出有意义的改进

### 5.6 Description 优化

技能 description 是触发的关键。优化流程（来自 skill-creator）：

1. **生成 20 个触发评估查询**：8-10 个应触发 + 8-10 个不应触发
2. **查询要具体真实**：包含文件路径、项目上下文、公司名、URL、用户背景故事
3. **反例要棘手**：关键词相近但实际需要不同技能的 near-miss
4. **迭代优化**：评估当前 description 的触发率，提出改进，重评估，最多 5 轮

```markdown
# ❌ 太抽象的查询
"Format this data"

# ✅ 具体真实的查询
"ok so my boss just sent me this xlsx file (its in my downloads, called something like 
'Q4 sales final FINAL v2.xlsx') and she wants me to add a column that shows the profit 
margin as a percentage. The revenue is in column C and costs are in column D i think"
```

### 5.7 检查清单验证

完成开发后，用第八章的检查清单逐项验证。未通过的项目需要修复后重新测试。

## 六、Token 效率

每一行都是持续的 token 开销。只保留必要内容。

### 6.1 量化目标

| 技能类型 | 主文件目标 | 说明 |
|---------|----------:|------|
| 阶段命令 | < 300 行 | 含骨架 + 就近约束，脚本外置 |
| 频繁加载的辅助技能 | < 150 行 | 只保留核心流程 |
| 其他技能 | < 200 行 | — |
| 子文件（references/） | 不限 | 按需加载，不占主文件 token |

> **出处说明：** 上表 < 300 行是项目自定激进目标，非外部标准。真正的平台建议来自：
> - **Anthropic 官方** (anthropic-best-practices.md): "Keep SKILL.md body under 500 lines for optimal performance"
> - **skill-creator** (SKILL.md): "<500 lines ideal"，另外 "For large reference files (>300 lines), include a table of contents"（300 是参考文件的"大文件"阈值，不是主文件限制）
>
> 即官方硬建议是 **500 行**，本项目的 300 行是对阶段命令更严格的自律要求。如果闸门密度需要，允许接近 500 行——防御密度优先于行数指标。

SKILL.md 保持 < 500 行 / 5000 token。超过时增加层级并明确指向下一级文件。

### 6.2 CSO（Claude 搜索优化）

让 Agent 在需要时能找到你的技能：

1. **description 只写触发条件** — 不写流程概要。测试发现写流程时 Agent 直接按概要执行，跳过正文。这是最重要的 CSO 规则。

2. **关键词覆盖** — description 里包含错误信息、症状、同义词、工具名，用 Agent 会搜的词

3. **描述性命名** — 动名词优先：`creating-skills` > `skill-creation`

4. **description 稍微"推"一点** — Claude 倾向于"欠触发"（该用技能时不用），可以在 description 中主动扩展适用场景

```markdown
# ❌ 被动——可能不触发
How to build a simple dashboard to display data

# ✅ 主动——覆盖更多触发场景
How to build a simple dashboard. Use this skill whenever the user mentions dashboards, 
data visualization, metrics, or wants to display any kind of data, even if they don't 
explicitly ask for a 'dashboard.'
```

### 6.3 Gotchas 是最高价值内容

环境特定、违反直觉的事实是技能最有价值的内容。Agent 知道通用知识，不知道你的项目特定陷阱。

```markdown
# ✅ Gotchas——Agent 不知道的陷阱
- 本项目的测试必须用 `npm run test:integration`，不是 `npm test`
- Node 18 的 fs.cp 有 bug，用 fs.cpSync 替代
- `config.local.yaml` 会被 git 忽略，但 `config.yaml` 不会
```

### 6.4 输出格式模板

比文字描述格式更可靠。Agent 对具体结构模式匹配能力强。

```markdown
# ❌ 文字描述格式
审查结果应包含文件名、问题描述和严重程度

# ✅ 模板——结构明确，无歧义
审查结果格式：
[文件:行号] 严重程度 | 描述
例：[src/auth.ts:42] HIGH | 硬编码密钥
```

### 6.5 压缩技巧

- **快速版 + 引用完整版** — 主文件 3 行快速版，子文件完整版
- **交叉引用替代重复** — 不重复其他技能的流程，写"详见 [skill-name]"
- **推到 --help** — 不在技能文件中列出所有 flag，写"支持多种模式，详见 --help"
- **一个优秀示例胜过多个平庸的** — 选最相关语言，完整可运行，注释 WHY
- **验证：** `wc -l <skill-path>/SKILL.md`

## 七、反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|---------|
| **MUST 堆砌** | 指令疲劳，Agent 选择性跳过看似严厉实则无区别的规则 | 解释 WHY；只在闸门型关键点用绝对语言 |
| **description 写流程概要** | Agent 直接按概要执行，跳过正文细节 | description 只写触发条件 |
| **全局约束与操作分离** | 约束集中在顶部/底部，步骤中不可见，Agent 做到那步时约束不在注意力范围内 | 约束拆散嵌入各步骤的验证门 |
| **单体内联** | 所有内容塞一个文件，改一处要检查多处；Token 消耗是常数，与场景复杂度无关 | 脚本外置到 references/，主文件保留快速版 |
| **叙事式示例** | "在 session X 中我们发现了..."是一次性叙事，不是可复用技术 | 写可复用的模式和技术 |
| **技能网而非链** | 技能间循环依赖，Agent 不知走哪条路 | 线性链条，终端状态指向唯一后继 |
| **空泛 description** | 缺乏触发关键词，技能不触发 | 包含错误信息、症状、同义词 |
| **技能触发太频繁** | description 太宽泛 | 更具体或加 `disable-model-invocation: true` |
| **压缩后失效** | 技能内容在上下文压缩后被裁剪 | 强化 description 和指令；用 hooks 确定性行为 |
| **Gotchas 缺失** | Agent 反复犯同样的项目特定错误 | 将纠正加入 Gotchas 段落 |
| **选项过多无默认** | Agent 在无效步骤浪费时间 | 提供默认方案，简要提及替代 |
| **过拟合特定案例** | 技能只在测试用例上工作，不可泛化 | 找更好的模式和比喻，不加 ad-hoc 规则 |

## 八、检查清单

### 结构

- [ ] description 只写触发条件，不写流程
- [ ] 有 Iron Law（核心铁律，代码块格式，2-3 行）
- [ ] 主文件 < 300 行（阶段命令）或 < 200 行（其他）
- [ ] SKILL.md < 500 行 / 5000 token
- [ ] 脚本提取到子文件，主文件保留快速版
- [ ] 全局约束拆散嵌入各步骤，不集中在顶部/底部

### 内容

- [ ] 解释 WHY，不过度使用 MUST（闸门型除外）
- [ ] 关键步骤有反例（什么算做得不够）
- [ ] 闸门型技能有抗合理化三层防御
- [ ] 调用外部技能使用 Skill 工具，不内联重写
- [ ] 引用使用命名空间格式 + 强度标记
- [ ] 有 Gotchas 段落（项目特定陷阱）
- [ ] 输出格式用模板而非文字描述

### 可靠性

- [ ] 闸门型步骤用绝对语言，路径型解释 WHY
- [ ] 关键决策点有 Gate Function 伪代码
- [ ] 每个步骤有幂等检查，支持断点恢复
- [ ] 技能链线性，终端状态指向唯一后继

### Token 效率

- [ ] 主文件行数在目标范围内（`wc -l` 验证）
- [ ] description 包含关键词（错误信息、症状、同义词）
- [ ] 无重复内容（交叉引用替代）
- [ ] 子文件引用最多一层深
- [ ] 重复逻辑捆绑为脚本

### 验证

- [ ] 基线测试（无技能观察 Agent 如何失败）
- [ ] 按类型选测试方法（压力场景/应用场景/检索场景/识别场景）
- [ ] 至少一轮 execute-then-revise 迭代
- [ ] Description 经过触发测试验证
