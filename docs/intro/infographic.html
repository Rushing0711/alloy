为什么不裸用 AI Agent？

三个根本问题，Alloy 逐个解决

01
需求漂移
Agent 理解的"要做的东西"和你想要的不一样，聊着聊着范围就跑了。没有 spec 约束，需求只在聊天记录里。
02
质量靠自觉
TDD、代码审查、分支隔离全看 Agent 当天心情。前脚记得后脚忘，没有硬性闸门强制质量流程。
03
改完不留痕
代码改了，但"为什么这么改"、"变了哪些 spec"没记录。换人（换 session）就断片，无法追溯决策链条。

6 个核心特性

不是建议，是硬约束

01
流程闸门，不是建议
跳过 TDD 被脚本阻断，跳过审查 hash 对不上。5 阶段每步硬校验。
02
OpenSpec + Superpowers
两个工具编成一条流水线，不用学 20+ 个技能，一条命令走到交付。
03
随便打哪个命令，都能接上
掉线回来打错命令也能自动路由到正确位置，不需要记住上次做到哪。
04
改需求不乱
做到一半改需求？还没写就回溯修正，已经写了就开新 change。规格代码不分叉。
05
每次做完，下次更聪明
每次 change 结束自动复盘，教训跨周期传递到 memory，下次自动提醒。
06
每一步都可追溯
每个制品独立 hash 锁定 + 独立 commit。谁在什么时候确认了什么，完整可查。

Bug 修复：诊断先行，分支后置

3 条修复路径，根据 change 状态自动分流

SCENE 1
Worktree 内修复
phase=applied 且 worktree 存在。在隔离环境中 systematic-debugging → TDD → 验证 → 精确提交。
SCENE 2
Feature 分支修复
applied(无 worktree) 或 planned。在 feature 分支上修复，提交时注明归属 change。
SCENE 3
Hotfix 热修分支
无活跃 change 或已 finish。创建 hotfix/ 分支 → TDD → 精确文本确认 → --no-ff 合回主分支。

[HARD STOP] 禁止在主分支上直接修改代码 — 一行和千行的保护等级一样
