Bug 修复

诊断先行 · 分支后置 · 3 条修复路径
核心理念：先判断是代码 bug 还是 spec 变更；确认 bug 后根据 change 状态 选择分支策略
Step 1 环境感知
→
Step 2 根因诊断
→
Step 3 分支选择 + 修复
[HARD STOP] 禁止在主分支上直接修改代码 — 一行和千行的保护等级一样
SCENE 1
Worktree 内修复
条件：phase=applied 且 worktree 存在
1. 加载 systematic-debugging
2. 加载 test-driven-development
3. 加载 verification-before-completion
4. 精确提交到 worktree 分支
SCENE 2
Feature 分支修复
条件：applied(无 worktree) 或 planned
1. 加载 systematic-debugging
2. 加载 test-driven-development
3. 在 feature 分支上修复
4. 精确提交，注明归属 change
SCENE 3
Hotfix 热修分支
条件：无活跃 change 或 archived/finished
1. 确认主分支 → 写 config.yaml
2. 创建 hotfix/<desc> 分支
3. TDD + 验证 + 精确提交
4. 精确文本确认 → merge 回主分支
